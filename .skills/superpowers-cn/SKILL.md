# 软件开发方法论工作流助手（Superpowers）

## 前置条件

本 Skill 为纯指令型，不依赖外部 API、MCP 或 CLI 工具。它是一套给编码 Agent 用的**完整软件开发方法论**，核心价值是**流程纪律**：写代码前先澄清、实现时先测试、调试时先定位根因、宣称完成前先拿证据。

> 优先级铁律：**用户显式指令 > 本方法论技能 > 默认系统行为**。用户指令说的是"做什么"，不代表"跳过流程"；但若用户明确要求跳过某流程（如"别用 TDD"），遵从用户——用户永远掌控。
>
> 技能分两类：**刚性**（TDD、系统调试——严格照做，不得把纪律调整掉）与**柔性**（模式类——把原则适配到情境）。

## 核心能力

### 一、技能系统总则

**用户触发**：任何开发任务开始时

**要点**：
1. 动手前先判断"有没有技能适用"，哪怕只有 1% 可能也要先查（澄清提问也算任务，查在提问前）
2. 采用某技能时宣告"正在用 X 技能做 Y"
3. 有清单的技能，为每一项建 TodoWrite 任务并按序完成
4. 识别红旗念头（"这太简单"、"就这一次跳过"），见 references/workflow-details.md 第 9 节

### 二、头脑风暴与设计澄清

**用户触发**："我想做个…"、"帮我加个功能"、"设计一下"

**硬门禁**：在提出设计并获用户批准前，不写代码、不搭脚手架、不调用实现技能。每个项目都走，包括"简单"的。

**流程**：探索上下文 → （视觉话题先单独提议可视化伴侣）→ 一次一个澄清问题 → 提 2-3 方案+推荐 → 分节呈现设计逐节获批 → 写设计文档 → 规范自查 → 用户复审 → 转入写计划。

### 三、写实现计划

**用户触发**："写个实现计划"、"把方案拆成任务"

**流程**：先规划文件结构（单一职责）；把工作拆成 2-5 分钟粒度任务，每个任务给确切文件路径、完整代码、验证步骤；贯穿 DRY/YAGNI/TDD/频繁提交。多子系统需先拆成独立子项目计划。

### 四、执行计划

**用户触发**："按计划执行"、"开始实现"

**流程**：带检查点的批量执行；每完成一批在人类检查点确认；严格按计划走，偏离即停。

### 五、子代理驱动开发

**用户触发**："用子代理开发"、"派代理逐任务实现"

**流程**：每个任务派新子代理，两阶段评审——先查规范符合度，再查代码质量；通过才推进下一任务。

### 六、并行子代理调度

**用户触发**："并行处理"、"同时跑多个任务"

**要点**：面对 2+ 个无共享状态、无顺序依赖的独立任务时，并行派发子代理；有依赖或共享状态则串行。

### 七、Git Worktree 隔离工作区

**用户触发**："隔离开发"、"开个 worktree"

**流程**：开始需隔离的特性工作或执行计划前，用原生工具或 git worktree 建立隔离工作区，跑项目初始化，验证测试基线干净。

### 八、测试驱动开发（TDD）

**用户触发**：实现任何特性或修 bug 前

**铁律**：没有失败的测试，就不写生产代码。红-绿-重构：写失败测试 → 亲眼看它失败 → 写最小代码通过 → 看它通过 → 提交。代码先于测试写了就删除重来。细则见 references/workflow-details.md 第 1 节。

### 九、系统化调试

**用户触发**：遇到任何 bug、测试失败、异常行为时

**铁律**：没完成根因调查，不许提修复。四阶段：根因调查 → 假设验证 → 最小修复 → 验证。症状式修复就是失败。细则见第 2 节。

### 十、完成前验证

**用户触发**：准备宣称"完成/已修复/通过"、提交或建 PR 前

**原则**：证据先于断言。必须真正运行验证命令并确认输出干净，再做任何成功声明。禁止无证据说"应该好了"。清单见第 6 节。

### 十一、请求代码评审

**用户触发**：完成任务、实现大特性、合并前

**流程**：对照计划评审，按严重度（Critical/Major/Minor）报告问题；Critical 阻断推进。

### 十二、接收代码评审

**用户触发**：收到评审反馈、准备照改前

**原则**：以技术严谨回应，不做表演式认同，也不盲目照改；反馈不清或存疑时先核实再实施。

### 十三、完成开发分支

**用户触发**：实现完成、测试通过后

**流程**：呈现结构化选项——合并 / 提 PR / 保留分支 / 丢弃；确认后清理 worktree，不擅自决定合并策略。

### 十四、编写技能（元技能）

**用户触发**："写个 skill"、"改进技能"

**流程**：按最佳实践创建/编辑技能，描述要精准触发，语气与纪律强度匹配技能类型；部署前用子代理测试是否按预期触发与执行。

## 执行优先级（严格约束）

1. **用户指令优先**：用户显式指令高于一切技能纪律。
2. **门禁不可跳**：设计需批准、测试需先失败、修复需先定位根因、完成需先验证——这些门禁不因"简单"或"赶时间"而豁免。
3. **刚性技能不放水**：TDD 与系统调试严格照做，违反字面即违反精神。
4. **证据胜过断言**：任何"完成/通过"声明必须有真实运行输出支撑。
5. **YAGNI / DRY / TDD 贯穿**：只做当前需要的，去重复，测试先行。
6. **宣告与追踪**：采用技能时宣告，有清单则建任务逐项完成。

## 沟通规则

- 默认使用中文回复。
- 采用某技能时先宣告"正在用 X 技能做 Y"。
- 有清单的流程，逐项建任务并按序完成。
- 以流程结论开头，再展开细节。
- 每步严格照流程门禁走，不跳步。

## 常见坑

1. "这太简单不需要设计/测试"——简单项目恰是未审视假设造成最多返工之处。
2. "我之后补测试"——事后测试立即通过，证明不了任何事。
3. "先改这一个地方就好"——症状式修复会掩盖根因、制造新 bug。
4. "应该修好了"——没跑验证不许宣称完成。
5. "删掉几小时工作太浪费"——沉没成本谬误，留不可信代码才是技术债。
6. "TDD 太教条我要务实"——TDD 就是务实，先测比事后调试快。
7. 把用户的"做什么"当成"跳过流程"的许可——用户没让你跳流程时不要自作主张跳。
8. 技能纪律与用户明确指令冲突时——遵从用户，用户永远掌控。
