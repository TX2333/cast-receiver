# Superpowers 工作流纪律细则参考

> 本文件浓缩 Superpowers 各流程的纪律要点、门禁与红旗清单，供主 SKILL.md 引用。核心哲学：测试先行、系统化胜过临时起意、复杂度最小化、证据胜过断言。

---

## 1. 测试驱动开发（TDD）细则

### 铁律
```
没有失败的测试，就不写任何生产代码。
```
先写测试 → 亲眼看它失败 → 写最小代码让它通过。如果没看着测试失败，就不知道它测的是不是对的东西。

### 红-绿-重构循环
1. **RED 写失败测试**：一个测试只测一个行为，命名清晰，用真实代码（除非不得已才用 mock）。
2. **验证 RED（强制）**：运行测试，确认它**失败**（而非报错），失败原因是"功能缺失"而非笔误。测试立即通过 = 你在测已有行为，需修正测试。
3. **GREEN 写最小代码**：只写刚好让测试通过的代码，不加额外特性、不顺手重构、不"改进"。
4. **验证 GREEN（强制）**：测试通过、其他测试仍通过、输出干净（无错误/警告）。
5. **REFACTOR 重构**：仅在绿灯后，去重复、改命名、抽helper，保持绿灯，不加新行为。

### 若代码先于测试写了
删除。重新开始。不留作"参考"、不"改编"、不看它。删除就是删除。

### 常见借口对照表
| 借口 | 现实 |
|------|------|
| "太简单不用测" | 简单代码也会坏，测试只花 30 秒 |
| "我之后补测试" | 事后写的测试立即通过，证明不了任何事 |
| "我已经手动测过了" | 临时 ≠ 系统，没记录、不能复跑 |
| "删掉几小时的工作太浪费" | 沉没成本谬误，留着不可信的代码才是技术债 |
| "TDD 太教条，我要务实" | TDD 就是务实：先测比事后调试快 |
| "测试难写 = 需求不清" | 听测试的话，难测 = 难用，简化接口 |

## 2. 系统化调试细则

### 铁律
```
没有完成根因调查，就不许提出任何修复。
```
症状式修复就是失败。随机试错浪费时间还制造新 bug。

### 四阶段（必须逐阶段完成才能进入下一阶段）
1. **Phase 1 根因调查**：仔细读错误信息与完整堆栈（常含确切答案）；稳定复现；定位到具体行/文件/错误码。
2. **Phase 2 假设与验证**：形成关于根因的假设，用最小实验验证，而非直接改。
3. **Phase 3 最小修复**：针对根因做最小改动，不顺手改别的。
4. **Phase 4 验证**：确认问题消失、无回归、输出干净。

### 配套技术
- **根因追踪**：从症状反向追到源头，别停在第一个能改的地方。
- **纵深防御**：在多个层次加校验，但先定位根因。
- **基于条件的等待**：用条件轮询替代 sleep，消除时序不稳定。

发现 bug 时先写一个能复现它的失败测试，再走 TDD 循环——测试既证明修复也防回归。

## 3. 头脑风暴与设计门禁

### 硬门禁
```
在提出设计并获用户批准前，不写任何代码、不调用任何实现技能、不搭脚手架。
```
每个项目都要走这个流程，包括"看起来很简单"的。简单项目恰恰是未经审视的假设造成最多返工的地方。设计可以短，但必须呈现并获批准。

### 清单（逐项建任务，按序完成）
1. 探索项目上下文（文件、文档、近期提交）
2. 需要视觉内容时先单独提议"可视化伴侣"
3. 一次只问一个澄清问题（优先多选）
4. 提出 2-3 个方案 + 权衡 + 推荐
5. 分节呈现设计，每节获批准后再继续
6. 写设计文档并提交
7. 规范自查（占位符/矛盾/歧义/范围）
8. 用户复审书面规范
9. 转入 writing-plans 技能

## 4. 写计划与执行

### 写计划（writing-plans）
- 假设执行者是技能不错但完全不懂本项目、品味存疑、抗拒测试的初级工程师。
- 每个任务拆成 2-5 分钟粒度的单一动作：写失败测试 → 跑它确认失败 → 写最小实现 → 跑测试确认通过 → 提交。
- 每个任务给出确切文件路径、完整代码、验证步骤。DRY、YAGNI、TDD、频繁提交。
- 先规划文件结构：每个文件单一职责，一起变化的放一起。

### 执行计划（executing-plans / subagent-driven-development）
- 每个任务派新子代理，两阶段评审：先查规范符合度，再查代码质量。
- 批量执行时设人类检查点。
- 2+ 个无共享状态、无顺序依赖的独立任务 → 用并行子代理。

## 5. 代码评审

### 请求评审（requesting-code-review）
完成任务、实现大特性、合并前，对照计划评审，按严重度报告问题。严重问题阻断推进。

**严重度分级：**
- **Critical**：正确性/安全/数据丢失风险，必须修复才能继续。
- **Major**：明显偏离计划或设计缺陷，应修复。
- **Minor**：风格/可读性，可择机处理。

### 接收评审（receiving-code-review）
以技术严谨回应，不做表演式认同，也不盲目照改。反馈不清或技术存疑时，先核实再实施。

---

## 6. 完成前验证（verification-before-completion）

### 原则
```
证据先于断言，永远如此。
```
在声称"完成/已修复/通过"之前，必须真正运行验证命令并确认输出，再做任何成功声明。禁止在没跑验证的情况下说"应该好了"。

### 清单
- [ ] 运行了相关测试/构建，并贴出真实输出
- [ ] 输出干净（无错误、无警告）
- [ ] 覆盖了边界与错误情况
- [ ] 声明的每一项都有对应证据

---

## 7. 完成开发分支（finishing-a-development-branch）

实现完成、测试通过后，呈现结构化选项：合并 / 提 PR / 保留分支 / 丢弃。确认后清理 worktree。不擅自决定合并策略。

---

## 8. 编写技能（writing-skills，元技能）

创建/编辑技能时遵循最佳实践：描述精准触发；内容可被子代理测试验证；语气与纪律强度匹配技能类型（刚性 vs 柔性）。部署前用子代理测试技能是否按预期触发与执行。

---

## 9. 红旗 / 合理化对照表（识别自我欺骗）

这些念头意味着"停"——你在合理化跳过流程：

| 念头 | 现实 |
|------|------|
| "这只是个简单问题" | 问题就是任务，先查技能 |
| "我得先了解上下文" | 查技能在澄清提问之前 |
| "先探索一下代码库" | 技能会告诉你怎么探索，先查 |
| "这个不需要正式流程" | 有对应技能就用 |
| "跳过 TDD 就这一次" | 这就是合理化，删了重来 |
| "先改这一个地方" | 做任何事之前先查 |
| "我记得这个技能" | 技能会演进，读当前版本 |
| "症状修掉就行" | 没找根因就是失败 |
| "应该修好了" | 没跑验证不许宣称完成 |

### 优先级
```
用户显式指令（最高） > Superpowers 技能 > 默认系统行为（最低）
```
用户指令说的是"做什么"，不是"跳过流程"。但若用户明确说"别用 TDD"，则遵从用户——用户永远掌控。

### 技能类型
- **刚性**（TDD、调试）：严格照做，不得把纪律"调整"掉。
- **柔性**（模式类）：把原则适配到情境。
技能自身会说明属于哪一类。
