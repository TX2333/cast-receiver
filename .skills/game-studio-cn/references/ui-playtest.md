# 游戏 UI 与试玩 QA

## UI 默认策略

游戏世界放在 canvas 或 WebGL，文字密集 UI 放在 DOM。这样更容易获得清晰排版、响应式布局、可访问性、输入框、按钮状态和调试能力。

适合 DOM 的内容：

- HUD：血量、分数、计时、目标、资源
- 菜单：开始、暂停、设置、关卡选择、结果页
- 覆盖层：教程、对话、失败、胜利、确认框
- 复杂文本：任务说明、道具描述、日志、排行榜

## Playfield 保护

正常游玩时优先保护玩家视野：

- 初始屏几秒内要让人知道怎么开始玩。
- 默认只保留一个主要持久 HUD 区。
- 最多再保留一个小的次要状态区。
- 中央 playfield 不放常驻大面板。
- 下方中部尽量保持清晰，尤其是移动、跳跃、瞄准类游戏。
- 提示尽量短暂，结束后自动退场。

## HUD 布局建议

2D 常见布局：

- 左上：生命、护盾、资源
- 右上：分数、计时、目标
- 左下 / 右下：移动端虚拟按钮
- 中上：短目标提示或 wave 信息
- 中下：只放短暂交互提示，不放常驻大文本

3D 低 chrome 布局：

- 一个紧凑 objective chip
- 一个小 status surface
- 关键交互使用短暂 prompt
- 长任务和设置放暂停菜单
- 目标标签要处理遮挡、距离和重叠

## 菜单与覆盖层

菜单要覆盖完整状态流：

- 开始页：明确主动作和进入方式。
- 暂停页：继续、重开、设置、退出。
- 失败页：说明失败原因，提供重试。
- 胜利页：显示结果、奖励或下一关。
- 设置页：音量、画质、输入提示、必要的无障碍选项。

覆盖层出现时要明确是否暂停 simulation。如果不暂停，要避免遮挡玩家必须立即响应的内容。

## 试玩流程

每次可试玩增量都按以下顺序检查：

1. 启动游戏，确认第一个可操作画面。
2. 执行主动作：移动、跳跃、攻击、互动、建造、选择等。
3. 触发一次成功路径和一次失败路径。
4. 截取代表性画面：开始、游玩中、压力状态、暂停、失败/胜利。
5. 单独检查 UI 层：文字、按钮、遮挡、响应式。
6. 单独检查渲染层：动画、相机、碰撞、特效、性能。
7. 按严重程度输出问题，附复现步骤。

## 截图检查重点

看截图时不要只看是否“好看”，重点检查：

- 玩家下一步要做什么是否明显
- 主角、敌人、危险、目标是否可读
- HUD 是否遮挡关键区域
- 文字是否能在目标背景上读清
- 移动端或窄屏是否挤压玩法区
- 失败、受击、奖励反馈是否明确
- 3D 深度关系和目标距离是否清楚

## 浏览器 QA 清单

基础：

- 游戏能启动到有用状态
- 首屏不是长时间空白
- 主动作响应稳定
- 暂停、恢复、重开可用
- 错误状态有可读反馈

视口：

- 宽屏、窄屏、常见笔记本尺寸可用
- canvas 和 DOM HUD 对齐
- 缩放后点击区域仍正确
- 移动端横竖屏策略明确

2D：

- Sprite 没有锚点漂移
- Tilemap 碰撞层正确
- 相机不剧烈抖动
- 下方 UI 不遮挡落点或移动方向

3D：

- 相机不遮挡主目标
- 深度关系可读
- GLB 加载失败有提示
- 后处理关闭后问题是否消失
- WebGL context loss 有恢复或提示路径
- 低性能设备有画质或效果降级

## 常见 UI 问题

- HUD 信息太多，玩家不知道看哪里。
- 教程遮挡了玩家需要操作的对象。
- 3D 标签堆叠在一起，没有距离或遮挡处理。
- 失败页只说“失败”，没有告诉玩家原因。
- 暂停菜单恢复后输入状态残留。
- 移动端按钮和浏览器边缘手势冲突。
